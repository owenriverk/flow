# Design: Flow — satellite river-gauge text bot

Generated by /office-hours on 2026-06-27
Status: DRAFT
Mode: Builder

## Problem Statement
Whitewater kayakers on overnight trips or remote put-ins have no cell service and
no easy way to check current river flows. They carry satellite messengers (Garmin
InReach) or iPhones with satellite texting. Flow lets a paddler send one short
message with a gauge name and get back the current flow — so they can make go/no-go
and safety calls from the riverbank.

## What Makes This Cool
A cold, wet paddler at a remote put-in texts `gauley summersville` from an InReach
and gets back the live flow in seconds, over a satellite, for free. No app, no
signal, no login. It turns a $400 satellite messenger into a river-flow oracle.

## Constraints
- The user's message rides a slow, short, sometimes-paid satellite hop. The
  interaction must be **one message out, one message back**, terse, and hard to
  get wrong.
- InReach replies are limited to ~160 characters. Keep the reply tight.
- Free / near-free to run. Target $0/mo for v1.
- Built solo, for fun and for the paddling community.

## Premises (agreed)
1. **v1 delivery = email (InReach only).** Paddler sends an InReach message to a
   dedicated inbox; the bot reads inbound mail, replies to Garmin's per-message
   reply address, and Garmin delivers the reply to the device. No Twilio, no SMS
   registration, no per-message cost. **SMS is v2.**
2. **v1 reply = raw flow only.** Fields: gauge name, location, USGS site ID,
   flow (cfs) + stage (ft), timestamp. No runnable judgment yet.
3. **Lookup = curated short names → USGS site IDs**, plus raw 8-digit USGS IDs
   accepted directly. Specific named sections map to their gauge. Fuzzy matching
   is a later add.
4. **Data = USGS Water Services instantaneous-values API.** Free, no key.
   Discharge param `00060` (cfs), gauge height `00065` (ft).
5. **Hosted on Cloudflare** — Email Routing catches inbound mail, an Email Worker
   (TypeScript) runs the logic and replies via `message.reply()`. $0, push-based,
   no server to keep alive. (Chosen over Railway in /plan-eng-review.)
6. **A2P 10DLC registration** is deferred to the v2 SMS phase. Not needed for email.

## How the email path works
- Dedicated inbox: a Gmail address, or a domain address on **Cloudflare Email
  Routing** (both free).
- Paddler adds the address as an InReach contact and sends e.g. `gauley summersville`.
- Garmin emails the inbox **from a unique per-message reply address**. Replying to
  that email routes the reply back to the device — no phone number or web form needed.
- Bot parses the query, calls USGS, and **replies to the Garmin address** via SMTP.
- iPhone satellite is SMS/iMessage only (no email), so it is **out of scope for v1**
  by design — it is the reason SMS is the v2 unlock.

**Load-bearing assumption to verify first:** that replying to the Garmin email
actually reaches the device. Documented Garmin behavior, but prove it before building.

## Architecture (locked via /plan-eng-review)
- **Host:** Cloudflare Email Routing + Email Worker (TypeScript). Inbound mail
  triggers the Worker; reply via `message.reply()` (built with a MIME helper like
  `mimetext`, carrying correct `In-Reply-To`/`References` headers).
- **Prerequisite:** a domain managed by Cloudflare with Email Routing enabled.
- **Module boundaries** (single responsibility, thin Worker adapter):
  - `parseInbound(emailMsg)` → `{ query, replyTo }` — strips Garmin footer/tracking
    link, captures the exact reply address.
  - `lookupGauge(text)` → `siteId | null` — normalize (lowercase/trim/collapse
    spaces), exact alias match, else 8-digit raw ID passthrough, else null.
  - `usgs(siteId)` → `reading` — fetch IV API with an ~8s timeout.
  - `formatReply(meta, reading)` → string — guaranteed ≤160 chars, number first.
  - `handleQuery(text)` → `replyText` — channel-agnostic core (string in, string
    out) so v2 SMS is just a new adapter. No email objects leak into it.
- **Aliases as data:** one `aliases.json`, loaded once. Adding a run is a data edit.

```
InReach ──sat──▶ Garmin relay ──email──▶ Cloudflare Email Routing
                                              │ triggers
                                              ▼
                                        Worker email()
                                   parseInbound → handleQuery
                                   ( lookupGauge → usgs → formatReply )
                                              │
                                              ▼
                                   message.reply() ──▶ Garmin ──▶ device
```

## Failure modes (each needs a test AND a user-visible reply, never silence)
| Failure | Handling | What the paddler sees |
|---|---|---|
| USGS timeout (>8s) or API down | catch, fallback reply | "couldn't reach gauge data, try again" |
| Unknown/misspelled gauge | lookup returns null | "not found — try the USGS ID or check spelling" |
| Stage-only gauge (no 00060) | omit cfs, show ft | gauge height only |
| Garbage/empty message | handleQuery guard | "not found" help text |
| Reply would exceed ~160 chars | formatter truncates name, keeps number | flow value never lost |
**Critical rule:** on a satellite link a silent failure wastes the user's message +
reply credit. Every path returns *some* reply.

## Test plan (100% of paths before ship — write alongside code)
Unit: `lookupGauge` (alias/raw-id/unknown/normalization), `usgs` (happy/stage-only/
bad-id/timeout/malformed-json), `formatReply` (full/≤160-guarantee/missing-stage),
`parseInbound` (footer-strip/reply-addr-capture), `handleQuery` (4 branches).
Integration/E2E: one live USGS site; inbound-email mock → reply sent to captured
address; USGS-down → "try again" reply. The footer-parse and ≤160 tests are the two
that prevent silent field failures.

## NOT in scope (v1)
- SMS / Twilio / A2P 10DLC — v2 (the iPhone-satellite unlock).
- Runnable-judgment ("GOOD/LOW/HIGH") and flow ranges — v2; needs AW range data.
- Fuzzy name matching — v2; v1 is exact alias + raw ID only.
- Per-user saved gauges, multi-gauge digest, usage DB — deferred; no DB in v1.
- USGS response caching — nice-to-have, tracked as a TODO, not a v1 blocker.

## Domain
**boofdaddy.com** (chosen). Email Routing on Cloudflare; the texted address can be
short, e.g. `flow@boofdaddy.com`.

## Units
Native per region: US (USGS + CDEC) → **cfs / ft**, Canada (WSC) → **cms / m**. Each
`Reading` carries `dischargeUnit`/`stageUnit`; `formatReply` prints them (cfs whole,
cms to one decimal, stage to two). Rationale: Canadian big-water runs are talked about
in cms — forcing cfs mis-speaks to the exact audience those runs serve.

## Data sources
- **USGS** (`src/usgs.ts`) — Instantaneous Values API. US runs. cfs/ft native.
- **WSC** (`src/wsc.ts`) — Water Survey of Canada via MSC GeoMet OGC API. 5 runs
  (Nahanni, Babine, Chilko, Firth, Stikine). **Native cms/m, no conversion.**
- **CDEC** (`src/cdec.ts`) — California Data Exchange Center JSONDataServlet. Required
  for CA class V (USGS doesn't carry CA real-time). cfs/ft native. Per-station config:
  each alias carries `sensor` (20 flow, 76 inflow, 1 stage) and `dur` (H/D/E) because
  CDEC stations vary; `-9999` is the missing sentinel; obsDate is PST (pinned -08:00).
  Stations identified + live-verified via CDEC station search, not guessed.
- Routing is explicit: each alias carries `source` ('usgs' default | 'wsc' | 'cdec');
  raw ids route by format (all-digits → USGS, `\d\d[A-Z][A-Z]\d\d\d` → WSC; CDEC only
  via curated aliases since a 3-char code is too loose to detect safely). `handleQuery`
  dispatches to the matching fetcher.

Live smoke test: `LIVE=1 npx vitest run test/live.test.ts` hits all three real APIs.

WSC local time: the MSC API returns `DATETIME_LST` (local standard time with offset)
alongside the UTC `DATETIME`; we use LST so Canadian runs show local river time. Note
it's *standard* time, so during DST the clock reads ~1h behind the wall clock (the
instant and date are exact). Verified against a live `08CE001` response.

## Deferred / unresolved gauges
- **Magpie (CEHQ 073503)** — Quebec's CEHQ is a separate API; slots in as a fourth
  `source` when built.
- **Upper San Joaquin / Devils Postpile** — no live CDEC flow station found for the
  upper run (the Florence Lake USGS gauge is dead; valley stations don't represent it).
  Dropped rather than point at a misleading downstream gauge. Needs a real station code.

Multi-gauge runs simplified to one primary gauge in v1:
- Cataract → Cisco (09180500) only; true flow is Cisco + Green River (09315000) summed.
- Main Salmon → White Bird (13317000); "salmon shoup" (13307000) is a separate key.
- Green → split into "green jensen" (Lodore/Split Mtn) and "green river ut" (Deso/below).
Summed/combined gauges are a v2 feature.

## USGS API reference
- Endpoint: `https://waterservices.usgs.gov/nwis/iv/?format=json&sites=<ID>&parameterCd=00060,00065`
- Response: `value.timeSeries[]` — each series has `variable` (00060/00065),
  the latest `values[0].value[0].value` (the reading) and `dateTime`.
- Handle: site has no discharge sensor (stage only), stale/missing reading, bad ID,
  USGS API down/timeout.

## Curated alias format (start small, hand-curated)
```json
{
  "gauley summersville": { "site": "03189100", "name": "Gauley R",   "location": "Summersville, WV" },
  "green narrows":        { "site": "03451500", "name": "Green R",    "location": "Tuxedo, NC" }
}
```
Match logic: normalize inbound text (lowercase, trim, collapse spaces) → exact alias
match → else if it's an 8-digit number treat as a raw USGS site ID → else reply
"not found, try the USGS ID or check spelling."

## Reply format (≤160 chars)
```
Gauley R, Summersville WV
USGS 03189100
2,800 cfs / 4.21 ft
16:45 Jun 27
```

## Approaches Considered
### Approach A: Minimal email bot (CHOSEN)
Inbound email → small worker → USGS → reply email. Curated aliases in a committed
JSON file. No database, no Twilio. Ships in a weekend, $0/mo.
- Effort: S. Risk: Low. Reuses: USGS API, free email infra.

### Approach B: Seeded from American Whitewater + DB
Same plumbing, but pre-load hundreds of named sections from American Whitewater's
National Whitewater Inventory (section → gauge → runnable range) and add a small DB
for usage logs and per-user saved gauges. Sets up runnable-judgment and SMS cleanly.
- Effort: M. Risk: Med. Deferred until the pipe is proven.

### Approach C: Dual-channel email + SMS from day one
Build Twilio SMS alongside email so InReach and iPhone-satellite both work at launch.
Widest reach, but eats A2P 10DLC friction/cost upfront and doubles the debug surface
before demand is proven. Rejected for v1.

## Recommended Approach
**Approach A.** Get the cheapest, simplest version into real hands, validate that
(a) the satellite-email round trip works and (b) people actually text it. Then
graduate to B's data and C's SMS once it has earned the effort.

## Open Questions
- Does replying to the Garmin email reliably reach the device? (verify — assignment #1)
- Ambiguous/misspelled names in v1: a short "did you mean" list, or just a clean
  "not found"? (v1 leans "not found" + help text; fuzzy is v2.)
- Stage-only gauges (no cfs): show ft only, or also say "no flow sensor"?

## Success Criteria
- A paddler with an InReach can text a gauge name from a no-signal location and
  receive correct current flow in a single reply.
- Round trip works end to end over satellite at least once in the real world.
- 3–5 paddling buddies use it on a real trip and it doesn't embarrass you.

## Distribution Plan
No binary — it's a hosted service. Distribution is the inbox address plus a
one-paragraph "add this contact, text it a gauge name" instruction shared in paddling
forums / group chats. Deploy: push to Railway (or Cloudflare Workers) on merge.

## The Assignment
Before writing any code: **send a message from an InReach to a plain email inbox you
control, then reply to that email, and confirm the reply shows up on the device.**
Borrow a device if you don't have one. This 15-minute test validates the single
assumption the entire project rests on. If it works, you have a clear path. If it
doesn't, you've saved yourself a weekend. Second prerequisite: point a domain at
Cloudflare and enable Email Routing so the Worker has an address to receive on.
